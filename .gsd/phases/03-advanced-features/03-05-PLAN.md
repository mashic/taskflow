<objective>
Implement role-based access control (RBAC) using the Strategy pattern for permission logic.

Purpose: Enforce board-level permissions based on team member roles, demonstrate Strategy pattern
Output: Permissions module with role-specific strategy implementations and guard

**Design Pattern: Strategy Pattern (GoF) - RBAC Variant**
- **Definition:** Define a family of algorithms, encapsulate each one, and make them interchangeable.
- **Implementation:** PermissionStrategy interface with OwnerPermissionStrategy, AdminPermissionStrategy, and MemberPermissionStrategy. Each implements canRead(), canWrite(), canManageMembers(), canDelete() differently. PermissionsService uses a Map<BoardRole, PermissionStrategy> to select the right strategy.
- **Interview talking point:** "I used the Strategy pattern again for RBAC because each role has distinct permission rules. The strategy map lets me add new roles (like VIEWER) without modifying existing code - just add a new strategy class."
</objective>
